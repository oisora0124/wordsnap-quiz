# WordBank 再構成ロードマップ

## 結論

全面書き直しは行わない。現行のPages配備、ローカル二重保存、D1同期、クイズ、SRSを維持し、
語義モデル、出典、学習イベント、スケジューラ評価を追加型・影運用で段階導入する。

## 現行コードで確認済みの状態

| 領域 | 現状 | 再構成案との差分 |
|---|---|---|
| 配備 | Cloudflare Pages + Pages Functions + D1 | 維持する |
| 保存 | localStorage + IndexedDB + JSON + D1 gzip同期 | 復元世代と認証V2は未実装 |
| 競合 | `baseRev` / `stateRev` とD1 CAS | 維持する |
| 学習 | 決定論的な段階式SRS、回答速度判定、復習期日 | FSRSそのものではなく、比較ログもない |
| 語彙 | word / meaning / 品詞 / 推定CEFR / enrich | lemma / sense / provenance / JACET / WordNetはない |
| 履歴 | 正誤と時刻を単語ごとに直近50件 | response time等は判断に使うが履歴へ保存しない |
| 教材 | 辞書・Datamuse・任意AI、例文フォールバック | 出典・承認状態を統一スキーマで保持していない |
| テスト | API契約、公開物、秘密情報、主要安全条件 | ブラウザE2Eと純粋関数の単体テストが不足 |
| 実装 | 約1.6万行の単一HTML | 段階抽出が必要 |

## 導入順序

### Phase 0: 安全基盤（進行中）

- 公開物、API契約、秘密情報、HTML一致の自動検査を維持する。
- 不正な同期・JSON入力がHTMLや学習状態を汚染しないよう正規化する。
- 破壊的操作のUndo、2段階確認、後方互換を回帰検査する。

### Phase 1: 観測可能性

- 現行の正誤履歴を壊さず、追加型の `review_event` 候補スキーマを定義する。
- `response_time_ms`、出題形式、ヒント、誤答種別は、まず端末内・明示同意で影記録する。
- 収集目的、保持期間、削除方法を決めるまで外部送信しない。

### Phase 2: 語義コアの影モデル

- `lemma`、`sense`、`item`、`provenance` を現行 `word` の隣に追加できるスキーマとして設計する。
- 既存 `word -> meaning` は当面そのまま正本とし、変換結果を影データで比較する。
- WordNet、JACET等はライセンス、版、ID安定性を確認してから取り込む。
- 既存JSONを先に書き換えず、コピー→変換→件数・語義差分検証→切替の順にする。

### Phase 3: スケジューラ比較

- 現行SRSを本番経路として維持する。
- FSRS候補は同じ回答履歴から期日だけを影計算し、既存の出題順へ反映しない。
- 期日差、復習量、D7/D30保持率を確認してから段階的に切り替える。

### Phase 4: 教材QA

- 例文、和訳、誤答候補へ `source`、`confidence`、`approval_status` を付ける。
- 全件をルール検査し、曖昧・複数正解・品詞衝突だけをモデル審査へ送る。
- 生成モデル名ではなく、根拠と検査結果を保存する。

### Phase 5: 開発オーケストレーション

- `ORCHESTRATION_POLICY.md` のA〜Eで開発タスクをルーティングする。
- 利用者のブラウザ内へ多モデルルーターを常設しない。まず開発・バッチ・監査で効果を測る。
- 成功率、再実行、テスト、後工程バグを記録できる場合だけルーティングを学習させる。

### Phase 6: 段階的ソース分割

- 正規化、マージ、SRS、教材検査など純粋関数から抽出する。
- 1コミット1責務で、移動と仕様変更を同時に行わない。
- PWA版と単一HTML版を単一ソースから生成する。

## 先に実装しない項目

- 現行SRSからFSRSへの一括置換
- 既存JSONを破壊的にsense形式へ移行
- 出典・ライセンス未確認の語彙データ取込
- 同意・削除手段のない学習ログ送信
- 既存リンクを無効化する同期認証変更
- 全面UI書き直し

## 主要リスク

| リスク | 緩和策 |
|---|---|
| 語義分割で既存成績が別senseへ誤移行 | 影モデル、件数照合、人手サンプル、旧データ保持 |
| FSRS切替で復習期日が急変 | 影計算、上限、段階展開、旧スケジューラへ即時復帰 |
| ログ追加で個人情報が増える | 端末内既定、最小項目、同意、保持期限、削除導線 |
| 外部語彙データの権利・版が不明 | 取得前に利用条件と版を固定し、provenanceを必須化 |
| モデル多段化で手戻りとトークンが増える | 自動検査優先、単一モデル既定、異常時だけエスカレーション |
