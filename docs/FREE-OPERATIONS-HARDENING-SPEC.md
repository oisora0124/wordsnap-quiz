# WordBank 無料・小規模運用向け基盤改修仕様書

- 文書ステータス: 運用中（未完了項目あり）
- 対象リポジトリ: `wordsnap-quiz`
- 実装状況の基準コミット: `216173f`
- 想定利用者: 開発者本人、家族、友人、少人数の知人
- 想定配布形態: `https://wordbank.pages.dev` のPWA／ブラウザ利用
- 対象外: 課金、広告、App Store／Google Play公開、教師・生徒管理、大規模会員基盤

## 実装状況（2026-07-21）

既存利用者への影響を避けられる範囲として、次の項目は実装済み。

- 旧 `?w=`、任意形式の既存同期キー、旧プレーンD1行を維持したAPI契約テスト
- GET／PUT以外、巨大Content-Length、巨大gzip入力をD1アクセス・解凍前に拒否
- Tesseract.js 7.0.0の固定URLとSubresource Integrity（SRI）検証
- localStorageとIndexedDBの両方が書き込み失敗した場合だけ表示する保存警告
- JSON読込・全削除・強制同期前の端末内Undoと、30日trash／90日削除墓標
- 破損した同期状態を空状態として扱わず、500で停止するfail-closed API
- 学習属性の型・範囲正規化と、競合時の保守的な学習状態マージ
- Service Worker更新失敗時に旧版を保持するcore-first更新
- 例文問題を「出さない／一部／全部」の明示選択に限定し、既定値を「出さない」に固定
- 個人キーをRefererへ送らない`no-referrer`方針と、暗号学的乱数を要求する新規キー生成
- 外部辞書・翻訳・AI通信、PWA復元条件、保持期間の限界を設定画面へ明示
- ルート版／公開版HTML、PWA配布物、内蔵300語、AIモデル名、秘密情報混入を検査する公開前テスト
- 旧行、競合、破損、1万語gzip往復、手動復旧を検証するAPI契約テスト
- 同時初回PUTの原子性、キー切替後の遅延応答破棄、同期応答契約を実コードで検査する回帰テスト
- 同期・保存版生成の通信タイムアウトと、クイズCEFR条件の端末内復元
- 将来の語義中心化に向けた、実行系へ未接続のreview-event／lexical-shadowスキーマ

同期キーの認証V2、失効、完全なレート制限、サーバー過去版、長期休眠D1行の定期削除、
CI上のブラウザE2Eは未実装。これらは既存リンクをそのまま
無効化すると利用者を締め出すため、本仕様のコピー移行・APIバージョニング・復旧手順を
用意した別フェーズで実施する。今回の早期拒否は濫用時の資源増幅を抑えるが、匿名アクセス
そのものを防ぐ対策ではない。

## 1. 目的

本仕様書は、WordBankを無料・小規模で継続利用する場合でも必要となる基盤改修を定義する。

優先する成果は次の3点である。

1. 単語・学習履歴を失わず、誤操作や同期事故から復旧できる。
2. 機能追加時に既存データや主要機能を壊さない。
3. 将来PCや担当者が変わっても、リポジトリから環境を再構築・再公開できる。

将来のアプリ化・有料化にも流用できる設計とするが、有料化専用機能を先回りして実装しない。

## 2. 現状と維持すべき機能

現行版には、以下の基盤が既に存在する。原則として作り直さず、互換性を維持する。

- `localStorage` と IndexedDB の二重保存
- JSON形式の手動バックアップ／復元
- Service Workerによるオフライン起動
- Cloudflare D1へのgzip圧縮保存
- `stateRev`／`baseRev`による競合検出と原子的CAS
- 複数端末の状態マージ
- 他人の同期リンクを開いた場合の切替確認
- 同期キーの再発行
- AIキーを常時保存しない初期設定
- 古い学習データの正規化とスキーマバージョン管理

以下は今後も残る主な不足である。実装済みの項目は冒頭の「実装状況」を正とする。

- サーバーには原則として最新状態しかなく、同期済みの破損・誤削除を過去版へ戻せない。
- 新しい同期キーへ移行しても、古い同期先を完全には失効・削除できない。
- アプリ本体が約1.5万行の単一HTMLで、変更影響を分離しにくい。
- ルートと公開ディレクトリに同じ `index.html` を複製している。同期コマンドと公開前検査で
  不一致は防止しているが、将来は単一ソースからのビルド生成へ移行する。

## 3. 共通設計原則

### 3.1 データ保護

- 空状態・古い状態・不正な状態を、正常な最新状態へ自動上書きしない。
- 置換、復元、キー切替、全削除の前には復旧可能なチェックポイントを作る。
- 既存データの移行はコピーを先に作り、検証後に接続先を切り替える。
- 破壊的操作は対象・影響・復元方法を表示し、原則として2段階確認にする。

### 3.2 後方互換

- 現在のJSONバックアップを読み込める状態を維持する。
- 既存の `?w=` 同期リンクを、移行期間中は読み込めるようにする。
- データスキーマ変更時は `learningSchemaVersion` とは別に、アプリ状態全体の
  `stateSchemaVersion` を追加する。
- 新しいクライアントが書いた未知フィールドを、古いクライアントが消さないようにする。

### 3.3 秘密情報

- APIキー、復旧用秘密鍵、認証トークンをHTML、JSONバックアップ、ログ、テストfixtureへ含めない。
- 通常の共有リンクへ、所有者専用の削除・失効権限を含めない。
- URLに秘密を含める必要がある場合は、可能な限りクエリではなくフラグメントを利用し、
  サーバーログやRefererへ送らない。

### 3.4 小規模運用

- 外部の有料サービスを必須にしない。
- Cloudflareの無料枠内で運用できるよう、保存回数・履歴数・データ量に上限を設ける。
- 高度な管理画面より、JSONでの退避と明確な復旧手順を優先する。

## 4. 改修1: 「ツールを保存」の修正

### 4.1 目的

利用者が、壊れていないWordBank本体を単一HTMLとして保存できるようにする。

### 4.2 現状（修正済み）

以前の `downloadStandalone()` は、公開されていない `styles.css`、`cefr.js`、
`enrich.js`、`app.js` を取得して結合しようとしていた。現在は、クエリとフラグメントを
除いた配布元 `index.html` を取得し、テーマ画像をData URL化してPWA専用manifest参照を
外す方式へ修正済み。取得には60秒の上限があり、生成物のJavaScript構文と秘密情報混入を
公開前検査で確認している。

### 4.3 要件

- `SAVE-001`: ボタン操作で、正常に開けるHTMLファイルを生成できること。
- `SAVE-002`: 保存ファイルへ現在の単語、成績、同期キー、AIキーを埋め込まないこと。
- `SAVE-003`: 保存ファイルを開いた場合、新しい独立したローカルデータ領域を使うこと。
- `SAVE-004`: `file://` で制限される機能は、保存前と初回起動時に明示すること。
- `SAVE-005`: 生成物に未解決のローカル参照や存在しないJS/CSS参照がないこと。
- `SAVE-006`: 将来ソース分割した後も、同じビルド処理から単一HTMLを生成すること。

### 4.4 推奨実装

実装済みの短期対応では、サーバーから取得した未実行の配布用 `index.html` を保存する。
現在のDOMを `outerHTML` で直列化してはならない。DOMには入力中の値や一時的なUI状態が
含まれる可能性があり、秘密情報混入の原因になる。

ソース分割後は、ビルド処理で次の2種類を生成する。

- `dist/index.html`: PWA公開用
- `dist/standalone/wordbank.html`: 単一HTML保存用

### 4.5 受け入れ条件

- Chrome、Safariで保存が成功する。
- 保存したHTML内を検索しても、使用中の同期キーとAIキーが存在しない。
- 新しいブラウザプロファイルで開き、サンプル追加、クイズ、JSON出力ができる。
- オフライン状態で基本画面が開く。
- 保存機能のブラウザテストがCIで成功する。

### 4.6 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| 現在のDOMを保存してAPIキーや同期キーが混入する | 秘密情報漏えい | 配布元HTMLまたはビルド成果物だけを保存し、DOM直列化を禁止する |
| 外部画像・manifest・Service Worker参照が切れる | 保存版の表示崩れ・オフライン不可 | 単一HTML版では必要画像をData URL化し、PWA専用タグを除外または無害化する |
| `file://` の保存制限を利用者が理解できない | 保存したのにデータが消えたと誤認 | 起動時警告とJSONバックアップ導線を残す |
| 公開版と単一HTML版が別ロジックになる | 片方だけ古くなる | 同一ソース・同一テストから両方を生成する |

## 5. 改修2: 自動バックアップと過去版復元

### 5.1 目的

誤削除、壊れたJSON読み込み、同期競合、アプリ不具合によるデータ損失から、利用者自身が
過去の正常状態へ戻せるようにする。

### 5.2 対象データ

バックアップ対象は `appState` とする。以下は含めない。

- Gemini／GroqなどのAPIキー
- 同期用アクセストークン、所有者用復旧秘密鍵
- 例文生成の一時ロック
- Service Workerキャッシュ
- 再生成可能でサイズの大きい一時キャッシュ

### 5.3 要件

- `BACKUP-001`: サーバーへの正常な更新前に、上書きされる直前の状態を履歴へ保存すること。
- `BACKUP-002`: 直近5世代と、直近7日の日次スナップショットを上限として保持すること。
- `BACKUP-003`: 1利用者あたりの履歴容量上限を設け、上限超過時は古い履歴から削除すること。
- `BACKUP-004`: JSON読み込み、強制取得、強制送信、全削除、過去版復元の前に、端末内チェックポイントを作ること。
- `BACKUP-005`: 設定画面からバックアップ日時、単語数、デッキ数を確認できること。
- `BACKUP-006`: 復元前に差分概要を表示し、2段階確認を行うこと。
- `BACKUP-007`: 復元操作自体も新しい履歴として残し、復元の取り消しを可能にすること。
- `BACKUP-008`: 壊れた状態や現在より未知のスキーマは復元候補として適用しないこと。
- `BACKUP-009`: 履歴取得に失敗しても、通常の保存とローカル学習は継続できること。

### 5.4 推奨データモデル

概念上、D1に次の履歴を追加する。実際の名前はマイグレーション時に確定する。

```sql
CREATE TABLE state_revisions (
  sync_space_id TEXT NOT NULL,
  rev INTEGER NOT NULL,
  state TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  reason TEXT NOT NULL,
  PRIMARY KEY (sync_space_id, rev)
);
```

`state` は現行と同じgzip+base64マーカー形式を使用する。現行状態の更新と履歴保存は、
可能な限り同一トランザクションまたは同一の原子的バッチで処理する。

### 5.5 復元フロー

1. 履歴一覧を取得する。
2. 選択した履歴のメタデータと差分概要を表示する。
3. 現在状態を端末とサーバーの両方へチェックポイント保存する。
4. 選択した状態を正規化・検証する。
5. 新しいrevisionとして保存する。revision番号を巻き戻さない。
6. 全端末へ通常の同期として伝播させる。

### 5.6 受け入れ条件

- 誤って全単語を削除した後、直前状態へ復元できる。
- 復元後に「復元前の状態」へ戻せる。
- 2端末利用中でもrevisionが逆行しない。
- 履歴が壊れている場合は適用せず、現在状態を維持する。
- 履歴保存失敗時に現在の学習操作が停止しない。
- 履歴の保持数と容量上限が自動的に守られる。

### 5.7 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| 履歴保存と本体更新の片方だけ成功する | 復元不能または履歴不整合 | 原子的バッチを使用し、失敗時は本体更新を再試行する |
| 世代ごとに大容量stateを保存する | D1容量・無料枠を圧迫 | gzip継続、保持上限、日次間引き、利用者別容量上限を設ける |
| 復元でrevisionを巻き戻す | 他端末が古い状態を最新と誤認 | 内容だけ復元し、revisionは必ず増加させる |
| 壊れた履歴を適用する | 全端末へ破損が伝播 | サーバー・クライアント双方で形状とスキーマを検証する |
| 過去版に未知フィールドがない | 新機能のデータが消える | 復元前に互換移行し、移行不能なら拒否する |
| 利用者が誤った履歴を選ぶ | 意図しない巻き戻し | 単語数・日時・差分・復元理由を表示し、復元前スナップショットを必須にする |

## 6. 改修3: 同期キーの失効・削除

### 6.1 目的

同期リンクが漏えいした場合、古いリンクからの読み書きを停止し、必要に応じてサーバー上の
データを削除できるようにする。

### 6.2 現行方式の制約

現在の `?w=...` は、保存場所の識別子と読み書き権限を兼ねるBearer Tokenである。
同じ秘密を使って「通常同期」と「失効」を行う設計では、漏えいした相手も同じ権限を持つため、
確実な所有者確認ができない。

したがって、新方式では次の3要素を分離する。

1. `syncSpaceId`: 秘密ではない保存先識別子
2. `accessToken`: 通常の読み書き用。別端末への参加リンクに含められる
3. `ownerSecret`: トークン更新・全削除用。通常の共有リンクには含めない

### 6.3 要件

- `SYNCSEC-001`: サーバーはアクセストークンと所有者秘密鍵を平文保存せず、ハッシュを保存すること。
- `SYNCSEC-002`: 通常同期APIは有効なアクセストークンを要求すること。
- `SYNCSEC-003`: トークン更新・失効・全削除APIは所有者秘密鍵を要求すること。
- `SYNCSEC-004`: 所有者秘密鍵はIndexedDBへ保存し、JSONバックアップや通常共有リンクへ含めないこと。
- `SYNCSEC-005`: 所有者秘密鍵は復旧コードとして明示的に書き出せること。
- `SYNCSEC-006`: アクセストークン更新後、古いトークンを即時拒否すること。
- `SYNCSEC-007`: 全削除後、同じ旧識別子・旧トークンで保存領域を再作成できないこと。
- `SYNCSEC-008`: 失効済み識別子は、状態を持たない墓標として一定期間または永続保持すること。
- `SYNCSEC-009`: 通常の参加リンクは、可能なら `#join=...` フラグメントで表現し、
  API呼び出し時だけAuthorization Headerへ変換すること。
- `SYNCSEC-010`: 既存 `?w=` 利用者を、自動でデータ消失させず新方式へ移行できること。
- `SYNCSEC-011`: 所有者秘密鍵を失った場合の制約を明示すること。運営者による裏口復旧は設けない。

### 6.4 既存利用者の移行

1. 旧キーで最新状態を取得する。
2. 新しい `syncSpaceId`、`accessToken`、`ownerSecret` を端末内で生成する。
3. 新領域へ状態をコピーする。
4. 新領域を再取得し、単語数・デッキ数・revisionを検証する。
5. クライアントの接続先を新領域へ切り替える。
6. 復旧コードの保存を案内する。
7. 利用者が明示的に承認した場合のみ、旧キーを墓標化して旧状態を削除する。

移行途中で失敗した場合は旧領域を維持し、次回再試行できること。

### 6.5 受け入れ条件

- 通常共有リンクだけを持つ端末から、所有者操作を実行できない。
- トークン更新後、旧リンクのGET／PUTが認証エラーになる。
- 新リンクでは既存データを継続利用できる。
- 全削除後、旧トークンで同じ領域を再作成できない。
- 移行中に通信を切断しても、旧領域か新領域の少なくとも一方に完全な状態が残る。
- 移行テストで既存の `?w=` データが失われない。

### 6.6 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| 新旧認証の移行不具合 | 同期不能・データへアクセス不能 | コピー→検証→切替の順を固定し、旧領域を先に消さない |
| ownerSecret紛失 | 失効・削除が永久にできない | 初回に復旧コード保存を案内し、設定画面で保存状態を確認できるようにする |
| ownerSecretを共有リンクへ混入 | 漏えい相手が所有者権限を得る | URL生成関数と復旧コード出力を分離し、自動テストで混入を検査する |
| ハッシュ設計が弱い | DB漏えい時にトークンを推測される | 256bit以上の乱数を使用し、サーバーでは安全なハッシュ比較を行う |
| 削除済み領域を再作成できる | 古いリンクが再び有効化される | 失効墓標を残し、同一IDのINSERTを拒否する |
| URL形式変更でホーム画面起動が壊れる | PWAから同期できない | iOS／Androidのホーム画面追加を含む移行テストを行う |
| 認証追加で既存クライアントが同期を繰り返す | API負荷・エラー表示増加 | APIバージョンを分け、旧方式の終了日と更新案内を設ける |

## 7. 改修4: 自動テスト基盤

### 7.1 目的

データ消失、同期競合、旧データ移行、クイズ不正をリリース前に検出する。

### 7.2 テスト層

#### A. 単体テスト

- `normalizeState`／`normalizeWord`
- JSONインポート検証
- 状態マージと削除墓標
- streak、SRS、正答率
- 誤答候補の安全判定
- スキーマ移行
- バックアップ保持期限
- 認証トークンの解析

#### B. APIテスト

- 新規保存、取得、更新
- `baseRev`競合と409
- gzip／プレーン互換
- 不正JSON、巨大body、巨大展開データ
- 履歴作成と復元
- トークン更新、旧トークン拒否、全削除
- 失効済みIDの再作成拒否

#### C. ブラウザE2Eテスト

- 初回起動→単語追加→再読込
- JSON出力→全削除→JSON復元
- オフライン起動
- 2ブラウザコンテキストの同時編集
- Service Worker更新
- 保存版HTMLの起動
- 旧データからの移行
- 同期リンク切替確認

### 7.3 要件

- `TEST-001`: 通常のテストは外部AI・辞書APIへ通信しないこと。
- `TEST-002`: fixtureに実利用者データ、APIキー、同期キーを含めないこと。
- `TEST-003`: 時刻、乱数、ネットワーク応答を制御可能にすること。
- `TEST-004`: 本番D1をテストから参照しないこと。
- `TEST-005`: Pull Requestまたは公開前コマンドで必須テストを実行すること。
- `TEST-006`: 重要テスト失敗時はデプロイしないこと。
- `TEST-007`: 手動実機確認項目をリリースチェックリストとして残すこと。

### 7.4 最低限の合格基準

- データ保存・復元・同期・移行に関する必須テストがすべて成功する。
- Chromium系ブラウザのE2Eテストが成功する。
- Safari／iPhone、Android Chromeは主要フローを実機確認する。
- テスト失敗を無視して本番公開する手順が標準手順になっていない。

### 7.5 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| テストに合わせて実装を歪める | 本番と異なる挙動 | 本番コードの公開インターフェースを検査し、テスト専用分岐を最小化する |
| モックが実APIとずれる | 誤検出・見逃し | API契約fixtureを保存し、少数の任意ライブテストを別枠で用意する |
| E2Eが不安定 | 失敗が無視される | 固定時刻・固定乱数・ローカルAPIを利用し、待機をDOM状態ベースにする |
| fixtureへ秘密情報が入る | リポジトリから漏えい | ダミー形式を使用し、秘密情報スキャンをCIへ追加する |
| テスト整備のため大規模分割を先に行う | 回帰範囲が拡大 | 最初は現行関数を抽出してテストし、段階的にモジュール化する |

## 8. 改修5: 再現可能なCloudflareデプロイ

### 8.1 目的

新しいPCまたは空のCloudflare環境でも、リポジトリの手順だけで検証環境を構築し、
安全に本番更新できるようにする。

### 8.2 必要成果物

- `migrations/0001_initial.sql`: 現行 `states` テーブル
- 後続の変更を追加する連番マイグレーション
- `wrangler.toml` または `wrangler.jsonc`
- D1バインディング名と環境別設定
- `package.json` のビルド・テスト・ローカル起動・デプロイ前確認コマンド
- `README.md` の初回構築、更新、バックアップ、復旧手順
- `.env.example`。実際の秘密情報は含めない
- ステージング環境でのスモークテスト
- 本番公開前チェックリスト

### 8.3 要件

- `DEPLOY-001`: クリーンcloneから公開物を生成できること。
- `DEPLOY-002`: D1スキーマをゼロから作成できること。
- `DEPLOY-003`: マイグレーションを複数回実行しても二重適用しないこと。
- `DEPLOY-004`: 本番DBへ破壊的変更を行う前にバックアップを取得すること。
- `DEPLOY-005`: ステージングと本番で別DBを使用すること。
- `DEPLOY-006`: Service Workerのキャッシュ名を公開物の変更と整合させること。
- `DEPLOY-007`: 公開後に起動、保存、取得、オフライン起動を確認すること。
- `DEPLOY-008`: ロールバック時にDBスキーマとクライアント互換性を確認すること。

### 8.4 受け入れ条件

- READMEだけを読んだ状態でローカル起動とテスト実行ができる。
- 空のテストDBへマイグレーションを適用し、APIテストが成功する。
- ステージングで保存・復元・同期が成功する。
- 本番用IDや秘密情報がGit管理されていない。
- 公開手順と復旧手順を別の端末でも再現できる。

### 8.5 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| D1バインディングの取り違え | 本番データへテスト書込み | 環境別DB名を明示し、本番操作には確認ステップを入れる |
| マイグレーション失敗 | API停止・データ読込不能 | 追加型変更を優先し、事前バックアップとステージング検証を必須にする |
| 設定ファイルへ秘密をコミット | 認証情報漏えい | `.env.example`のみ管理し、秘密スキャンを行う |
| Service Workerが旧HTMLを返す | 新旧コード混在 | ビルドIDからキャッシュ名を生成し、更新E2Eを行う |
| クライアントだけ先に公開される | 新API未対応で同期停止 | サーバーを後方互換で先に公開し、その後クライアントを公開する |
| ロールバックで新データを読めない | データ欠落・同期停止 | 最低1世代前のクライアントとの互換テストを維持する |

## 9. 改修6: ソース分割と単一の生成元

### 9.1 目的

機能ごとの変更範囲を限定し、レビュー・テスト・将来のアプリ化を容易にする。

### 9.2 推奨構成

```text
wordsnap-quiz/
  src/
    index.html
    styles/
      app.css
    js/
      app.js
      state.js
      storage.js
      sync.js
      import.js
      quiz.js
      srs.js
      ai.js
      ui.js
  public/
    assets/
    wordsnap.webmanifest
    wordsnap-sw.js
  functions/
    api/
      wordsnap-state.js
  migrations/
  tests/
  scripts/
    build.mjs
  dist/                 # 生成物。直接編集しない
```

実際の移動先はCloudflare Pagesの設定に合わせて確定する。

### 9.3 要件

- `SOURCE-001`: 編集対象となる正本を `src/` に一本化すること。
- `SOURCE-002`: ルート版と公開版を手動コピーしないこと。
- `SOURCE-003`: `dist/` はビルドで再生成可能で、直接編集しないこと。
- `SOURCE-004`: モジュール間の依存方向を定義し、循環依存を避けること。
- `SOURCE-005`: グローバル状態への直接アクセスを段階的に減らすこと。
- `SOURCE-006`: 既存の関数名・保存キー・データ形式を一度に変更しないこと。
- `SOURCE-007`: 各抽出単位でテストを追加し、挙動同等を確認すること。
- `SOURCE-008`: PWA版と単一HTML版を同じソースから生成すること。

### 9.4 移行方法

全面書き直しは行わず、次の順で段階的に抽出する。

1. 定数、純粋関数、データ正規化
2. SRSとクイズ選択
3. JSON入出力とローカル保存
4. 同期と状態マージ
5. AI・辞書・OCR連携
6. DOM描画とイベント処理
7. CSSとHTMLテンプレート

各段階で、ビルド前後の主要画面とデータ出力が一致することを確認する。

### 9.5 受け入れ条件

- `npm run build` だけで公開物と単一HTML版を生成できる。
- 生成直後のGit差分が毎回同一で、時刻などの不要な差分が入らない。
- 既存のlocalStorageキーとJSONバックアップを継続利用できる。
- 保存、同期、クイズ、OCR、AIなしの例文モードが回帰していない。
- ルート `index.html` と公開版の不一致が構造的に発生しない。

### 9.6 変更リスクと緩和策

| リスク | 影響 | 緩和策 |
|---|---|---|
| スクリプト実行順が変わる | TDZ例外・起動不能・データ初期化 | 純粋関数から段階抽出し、起動E2Eとデータ保持テストを各段階で行う |
| 暗黙のグローバル依存を見落とす | 特定操作だけ失敗 | 参照検索とブラウザコンソール監視を行い、1機能ずつ移す |
| 大規模差分でレビュー不能になる | 回帰を見逃す | 1コミット1責務とし、全面整形を機能移動と同時に行わない |
| ビルド生成物と正本を両方編集する | 再びドリフト | 生成物へヘッダーを付け、CIで直接編集を検出する |
| `file://` とPWAで挙動が変わる | 保存版だけ利用不能 | 両成果物をE2E対象にする |
| Service Workerのパスが変わる | オフライン起動不能 | スコープと配信パスを固定し、キャッシュ更新テストを行う |

## 10. 実装順序

安全性を優先し、以下の順で進める。

### フェーズ0: ベースライン固定

- 現行版のJSONバックアップを取得する。
- 主要画面と主要操作の手動チェックリストを記録する。
- 保存、復元、同期の最小E2Eテストを作る。

### フェーズ1: 直近の壊れた機能を修正

- 「ツールを保存」を修正する。
- 秘密情報が保存ファイルへ入らないテストを追加する。

### フェーズ2: 再構築可能性を確保

- 現行D1の `schema.sql` とマイグレーション基盤を追加する。
- Cloudflare設定、ローカル起動、ステージング手順を文書化する。

### フェーズ3: データ復旧

- 端末内チェックポイントを追加する。
- サーバー履歴と復元UIを追加する。
- 保持期限・容量上限を検証する。

### フェーズ4: 同期認証V2

- `syncSpaceId`、`accessToken`、`ownerSecret` を導入する。
- 旧リンク移行を実装する。
- 失効、削除、旧ID再作成拒否を実装する。

### フェーズ5: 段階的ソース分割

- テスト可能な純粋関数から抽出する。
- PWA版と単一HTML版のビルドを一本化する。
- 手動コピー運用を廃止する。

## 13. 完了の定義

本仕様の基盤改修は、以下をすべて満たした時点で完了とする。

1. 保存版HTMLが秘密情報を含まず正常動作する。
2. 誤削除や壊れた同期から、利用者が過去状態へ復元できる。
3. 漏えいした同期リンクを失効できる。
4. 保存・同期・復元・移行の自動テストが公開前に実行される。
5. クリーンcloneからD1を含む検証環境を再構築できる。
6. PWA版と単一HTML版が単一のソースから生成される。
7. 現在のJSONバックアップと既存利用者データが維持される。

## 14. 実装時に別途決定する事項

以下は実装開始時に決定記録を残す。

- バックアップの正確な保持数、日数、利用者別容量上限
- 認証V2のAPIパス、Header名、トークンハッシュ方式
- ownerSecret紛失時の最終的な扱い
- 旧 `?w=` 方式の移行期間と停止条件
- Cloudflare D1のステージング／本番DB名
- 使用するテストランナーとブラウザE2Eツール
- `dist/` をGit管理するか、デプロイ時生成のみにするか
- ルート `index.html` の互換コピーを残す期間

決定はREADMEだけに埋めず、日付・理由・代替案・変更影響を短いADRとして記録する。
